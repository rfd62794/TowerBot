## ADR-033: MCP Compatibility

**Status:** Planned

**Context:**
Robert's workflow operates three systems in parallel: Claude (architect) → Windsurf (executor) → PrivyBot (operator). No direct connection exists between Claude and PrivyBot. During architecture sessions, Robert manually checks stats, copies context, and bridges the gap. MCP (Model Context Protocol) — Anthropic's open protocol — closes that loop. PrivyBot's 46 tools become callable from Claude directly.

**Decision:**

**Direction:** PrivyBot as MCP server — exposing tools TO Claude. Consuming MCP servers FROM PrivyBot is a separate future concern.

**Process model:** Standalone process, not embedded in the bot. MCP server lifecycle belongs to Claude Desktop, not the Telegram bot. A bot crash should not kill Claude's tool access. A separate `uv run python infra/mcp_server.py` starts the server independently.

**Transport:** stdio (Claude Desktop standard). Local only — no port, no network exposure, no authentication required. Security boundary is the local machine.

**Tool scope:** All 46 tools exposed. Tool descriptions already contain WHEN and DO NOT CALL guidance. Claude decides appropriateness at call time. Curating the list creates maintenance debt every time a tool is added.

**Single source of truth:** TOOL_REGISTRY remains the only definition. One converter function transforms OpenAI JSON Schema → MCP inputSchema at server startup. No duplicate tool definitions anywhere.

**Converter logic (mechanical):**
```python
# From TOOL_REGISTRY entry:
{
  "name": "get_channel_summary",
  "description": "...",
  "parameters": { "type": "object", "properties": {...} }
}

# To MCP tool:
mcp.Tool(
  name="get_channel_summary",
  description="...",
  inputSchema={ "type": "object", "properties": {...} }
)
```

**Claude Desktop configuration:**
```json
{
  "mcpServers": {
    "privybot": {
      "command": "uv",
      "args": ["run", "python", "infra/mcp_server.py"],
      "cwd": "C:/Github/PrivyBot"
    }
  }
}
```

**What this enables:** From any Claude conversation — "what are my YouTube stats this week?" → calls `get_channel_summary` directly against live data. Architecture sessions gain live data without context switching. The three-system loop closes.

**Implementation order:**
1. `pip install mcp` 
2. `infra/mcp_server.py` — MCP server wrapping TOOL_REGISTRY
3. Converter function: OpenAI schema → MCP inputSchema
4. Test with Claude Desktop
5. Claude Desktop config snippet in `README.md` 

**Consequences:**

*Positive:* Claude gains 46 live tools in every conversation. TOOL_REGISTRY is single source of truth — tools defined once, work everywhere. Zero impact on bot operation. Architecture sessions have live data.

*Negative:* Second process to start when Claude access is wanted. Write tools (save_commitment, add_personal_task) execute against the live DB when called from Claude — intentional but explicit. Requires Claude Desktop for stdio transport.

*Mitigations:* Server start is one command, can be added to Windows startup. Write tool side effects are covered by 204/204 tests. Claude Desktop is Robert's primary Claude interface.

**References:** ADR-007, ADR-028
