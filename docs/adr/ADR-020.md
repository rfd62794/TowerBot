# ADR-020: Preload on Startup Pattern

## Status
Accepted

## Context
PrivyBot's first query to any API incurs full latency (HTTP request, authentication, response parsing). For frequently-used APIs (weather, YouTube analytics), this adds noticeable delay to the first user interaction after bot startup.

The goal is to reduce first-query latency by warming the cache during bot startup, without significantly delaying bot availability.

## Decision
Implement a preloader that runs during bot startup to warm cache for frequently-used APIs.

### Preloader Design
```python
# scripts/preload.py
from tools.api._base import cached_api_call, make_params_hash
from core.db import record_preload_result

PRELOAD_TASKS = [
    # (tool_name, params, ttl_seconds, priority)
    ("weather_api", {"location": "Toronto"}, 3600, "high"),
    ("youtube_api", {"days": 7}, 21600, "high"),
    ("steam_api", {}, 86400, "normal"),
    ("gmail_api", {}, 300, "normal"),
    # ... more tasks
]

def run_preload(timeout_seconds: int = 30) -> dict:
    """
    Warm cache on startup.
    Returns: {"success": count, "failed": count, "skipped": count}
    """
    results = {"success": 0, "failed": 0, "skipped": 0}

    for tool_name, params, ttl, priority in PRELOAD_TASKS:
        params_hash = make_params_hash(**params)

        try:
            # Force fresh fetch (stale_ok=False)
            def live_fn():
                # Import and call the actual API function
                # This is a placeholder — actual implementation imports per tool
                pass

            result = cached_api_call(
                tool_name=tool_name,
                params_hash=params_hash,
                live_fn=live_fn,
                ttl_seconds=ttl,
                stale_ok=False,  # Force fresh fetch
            )

            if "error" in result:
                record_preload_result(
                    tool_name, params_hash, {}, ttl/3600,
                    False, 0, result.get("error")
                )
                results["failed"] += 1
            else:
                record_preload_result(
                    tool_name, params_hash, result, ttl/3600,
                    True, 0, None
                )
                results["success"] += 1

        except Exception as e:
            results["failed"] += 1

    return results
```

### Startup Integration
```python
# privybot.py
from scripts.preload import run_preload

def _initial_sync():
    """Run initial sync tasks on startup."""
    logger.info("Running initial sync...")

    # Preload cache (non-blocking with timeout)
    preload_result = run_preload(timeout_seconds=30)
    logger.info(
        f"Preload complete: {preload_result['success']} success, "
        f"{preload_result['failed']} failed"
    )

    # ... other initial sync tasks
```

### Timeout Strategy
- Total preload timeout: 30 seconds
- Individual task timeout: 5 seconds
- If timeout exceeded: skip remaining tasks, log warning
- High-priority tasks run first (weather, YouTube)

### Preload Task Selection Criteria
Include in PRELOAD_TASKS if:
- API is called frequently (daily or more)
- API has stable default parameters (e.g., weather for user's location)
- API latency is noticeable (> 500ms)
- Data has reasonable TTL (not too short to make preloading wasteful)

Exclude from PRELOAD_TASKS if:
- API is rarely called (once per week or less)
- API has no stable default parameters (requires user input)
- API latency is negligible (< 100ms)
- Data has very short TTL (< 5 minutes)

### Preload Logging
All preload attempts logged to `preload_log` table:
```sql
SELECT tool_name, fetched_at, success, duration_ms, error_msg
FROM preload_log
ORDER BY fetched_at DESC
LIMIT 10;
```

## Consequences
### Positive
- First-query latency reduced for frequently-used APIs
- User experience improved (faster responses)
- Cache warmed before first interaction
- Preload status visible via `get_preload_status()`

### Negative
- Startup time increased (30 seconds max)
- API quota used on startup even if no user query
- Preload failures may go unnoticed until user query
- Need to maintain PRELOAD_TASKS list

### Mitigations
- Timeout limits prevent excessive startup delay
- Preload status logged for monitoring
- Health check checks preload status
- PRELOAD_TASKS reviewed quarterly
- Preload failures logged to preload_log

## Examples

### Successful preload
```python
# Bot startup
[INFO] Running initial sync...
[INFO] Preloading weather_api (location=Toronto)...
[INFO] Preload complete: 5 success, 0 failed

# User query 1 minute later
result = get_current_weather("Toronto")
# Returns from cache (no API call)
# Latency: < 10ms (vs 500ms without preload)
```

### Partial preload failure
```python
# Bot startup
[INFO] Running initial sync...
[INFO] Preloading weather_api (location=Toronto)...
[WARN] Preload failed for youtube_api: API timeout
[INFO] Preload complete: 4 success, 1 failed

# User query for YouTube (first query)
result = get_channel_summary(days=7)
# API call made (preload failed)
# Latency: 500ms (normal)
```

### Timeout exceeded
```python
# Bot startup
[INFO] Running initial sync...
[INFO] Preloading weather_api (location=Toronto)...
[WARN] Preload timeout exceeded (30s), skipping remaining tasks
[INFO] Preload complete: 3 success, 0 failed, 2 skipped

# Bot starts despite incomplete preload
# Remaining tasks cached on first user query
```

### Preload status check
```python
# scheduler.py health_check
status = get_preload_status()
for entry in status:
    if entry["age_minutes"] > 60:
        logger.warning(
            f"Preload stale: {entry['tool_name']} "
            f"({entry['age_minutes']}min old)"
        )
```
