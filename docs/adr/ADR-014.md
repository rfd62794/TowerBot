# ADR-014: Data Retention and Cache Strategy

## Context
PrivyBot previously had no systematic data retention strategy. Every API call hit the network fresh, wasting quota and providing no historical context. The only historical storage was `channel_history` for daily channel stats. Cache existed but was inconsistently used — only YouTube tools used it, while game, recommendation, and search tools bypassed it entirely.

## Decision
Implement a three-layer data strategy:

### Layer 1: Cache (short-term, TTL-based)
- All 21 tools now use `tool_cache` table consistently
- TTL policy per tool based on data staleness:
  - Stable data (AppID resolution, wiki): 7 days
  - Semi-stable (game metrics, YouTube analytics): 24 hours
  - Dynamic (recommendations): 12 hours
  - Fast-stale (search, news, reddit): 30 min - 1 hour
  - Weather: 1 hour
- Prevents redundant API calls within TTL window

### Layer 2: Historical Storage (long-term, accumulates)
Five new tables added to `core/db.py`:

**video_history**
- Per-video daily stats: views, watch time, avg duration, avg retention
- Primary key: (video_id, date)
- Used for video performance tracking over time

**game_history**
- Per-game daily stats: players_2weeks, owners range, price, sale status
- Primary key: (appid, date)
- 90-day retention window
- Enables trend analysis and price tracking

**weather_history**
- Daily weather: temp, condition, wind
- Primary key: date
- Used for briefing context

**video_metadata_cache**
- Stable video data: title, description, tags, duration, published_at
- Primary key: video_id
- Updated on change, not time-based

**scheduled_videos**
- Scheduled YouTube videos with publish time
- Primary key: video_id
- Auto-cleanup after 7 days post-publish

### Layer 3: Context Enrichment
Tools return trend data when history exists:

**get_channel_summary**
- Returns `trend` dict with prior week comparison
- Views change %, subs change %
- `history_days_available` field

**get_game_metrics**
- Returns `players_prev_week` and `players_change_pct`
- `history_weeks_available` field
- Enables "Raccoin: 4,200 players (+10% vs last week)"

**Morning briefing**
- Enriched with game trend alerts (>10% change)
- Shows scheduled videos for today
- Channel stats with trend comparison

## Rationale

### Why separate cache from history?
- Cache prevents redundant same-day calls (performance)
- History accumulates for trend analysis (insight)
- Different purposes, different TTLs
- Cache expires, history persists

### Why these TTLs?
- **7 days**: AppID resolution and wiki are stable references
- **24 hours**: Game metrics and YouTube data change daily
- **12 hours**: Recommendations need freshness but not real-time
- **30 min - 1 hour**: Search/news/reddit stale fast
- **1 hour**: Weather changes slowly

### Why 90-day game history?
- Player trends visible over months
- Price tracking for sale alerts
- Reasonable storage footprint
- Can be extended later

### Why 7-day scheduled video cleanup?
- Prevents stale data accumulation
- Videos are published or deleted within days
- Reduces table size

### Why context enrichment?
- Agent sees trends, not just point-in-time
- Enables "up vs down" decisions
- Briefing becomes more actionable
- No extra API calls — uses existing history

## Implementation
- `core/db.py`: 5 new tables + CRUD functions
- `tools/games/metrics.py`: Cache wrappers for 6 functions + history recording
- `tools/recommendations.py`: Cache wrapper for get_content_recommendations
- `tools/search_tools.py`: Cache wrappers for 5 functions + weather history
- `tools/youtube.py`: Context enrichment for get_channel_summary
- `bot/scheduler.py`: Morning briefing enrichment (game trends, scheduled videos)
- `scripts/verify.py`: 6 new tests (3 tables, 2 cache, 1 history)

## Consequences
- All 21 tools now cache consistently (was 9/21, now 21/21)
- Historical data accumulates automatically
- Agent responses include trend context
- Morning briefing more informative
- Reduced API quota usage
- 30/30 tests pass
- Storage grows slowly (~1KB/day per tracked game)
