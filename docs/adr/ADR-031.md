# ADR-031: Missed Task Push-Forward and Recurrence Recovery

**Status:** Accepted

**Context:**
Personal tasks with `due_date < today` and `status='pending'` accumulated indefinitely. No mechanism existed to advance them or clean them. The morning briefing displayed these as active tasks, polluting the personal tasks section with stale obligations.

Two categories of missed task require different treatment:
- *Recurring tasks:* the specific instance is gone. Next occurrence should be computed from today, not the missed date.
- *Non-recurring tasks:* the obligation is real. Persistent pressure is appropriate until the user explicitly resolves it.

**Decision:**
Nightly job at 23:59 calls `push_missed_tasks()` as its first operation:

1. Query all `status='pending'` tasks where `due_date < today` 
2. Compute `new_date`:
   - Recurring: `next_recurrence_date(recurrence, anchor=today)` 
   - Non-recurring: `today + 1 day` 
3. Collision check against ADR-030 UNIQUE index:
   - If `(title, new_date)` already has a pending row → DELETE old missed row (future row covers the obligation)
   - Else → UPDATE old row's `due_date` to `new_date` 
4. Queue nudge per task regardless of collision outcome
5. Send batched nudge message after loop completes

`next_recurrence_date(recurrence: str, anchor: date) -> date` rules:
- `"daily"` → anchor + 1
- `"weekly:{day}"` → next occurrence of that weekday from anchor, minimum anchor + 1 (never returns anchor itself)
- Unknown format → anchor + 1, no exception

Nudge format (batched, one send per nightly run):
```
⚠️ Still pending: {title} (was due {old_date}).
Get it done or remove it from your list.
```

**Rationale:**

*Why push rather than delete:* Non-recurring missed tasks represent real obligations. Silent deletion is data loss. The user must explicitly decide to act or remove.

*Why anchor to today not missed date:* Anchoring to missed date cascades — a daily task missed 3 days generates 3 future occurrences. Anchoring to today collapses all misses to a single next occurrence.

*Why never-same-day rule:* If anchor is Monday and recurrence is `weekly:monday`, returning anchor means zero progress. Always advance by at least one day.

*Why batch nudges:* One Telegram send per nightly run regardless of miss count. Prevents notification floods.

*Why push before summary:* Nightly summary reflects post-push state. Summary should show what's actually pending tomorrow, not yesterday's stale list.

**Consequences:**

*Positive:*
- Morning briefing shows only relevant upcoming tasks
- Persistent nudge transfers responsibility to user without data loss
- Collision handling integrates cleanly with ADR-030 UNIQUE partial index

*Negative:*
- Non-recurring tasks pushed indefinitely until user acts — could become noise over time
- No mechanism to batch-dismiss stale obligations without reviewing each

*Mitigations:*
- User resolves via `/todo` command
- Future: `/dismiss {id}` command for explicit removal without completing (backlog)
