# ADR-022: BaseTool and BaseAPIHandler

## Status
Accepted

## Context
API clients in `api/*.py` and tool functions in `tools/*.py` each implement cache logic, error handling, and return shapes independently.

Problems:
- No enforced contract between tool files
- Each API file implements its own cache pattern
- Each tool file handles `stale_notice` and error shapes independently
- New tools require pattern knowledge not captured in code
- Inconsistent return shapes across tools
- No validation that required fields are present

## Decision
Create two base classes to enforce consistent patterns:

### BaseAPIHandler (`api/_handler.py`)
Base class for all `api/*.py` clients.

**Features:**
- `CACHE_PREFIX` attribute — must be set by subclass (validated at runtime)
- `cache_key(suffix)` — namespaced cache key generation
- `call(suffix, params_hash, live_fn, stale_ok)` — delegates to `CacheManager`
- `hash(*args, **kwargs)` — convenience wrapper for `cache.hash()`
- `_get_client()` — abstract method for credential loading

**Subclass pattern:**
```python
class WeatherAPIHandler(BaseAPIHandler):
    CACHE_PREFIX = "weather"

    def _get_client(self):
        return None  # Open-Meteo needs no auth

    def get_current_weather(self) -> dict:
        def _live():
            # HTTP call
            return {"temp_f": ..., ...}
        return self.call("current", self.hash(), _live)

# Module-level instance for backwards compat
weather_api = WeatherAPIHandler()

# Module-level function for backwards compat
def get_current_weather() -> dict:
    return weather_api.get_current_weather()
```

### BaseTool (`tools/_tool.py`)
Base class for all `tools/*.py` files.

**Features:**
- `success(data, stale_result)` — standard success return shape with `ok=True`, `stale_notice`
- `error(message, code)` — standard error return shape with `ok=False`, `error`, `error_code`
- `stale_notice(result)` — convenience wrapper for `cache.stale_notice()`
- Internal key stripping — removes `_stale`, `_cached_at`, etc. from data before returning

**Return shape enforcement:**
```python
# Success
{
    "ok": True,
    "stale_notice": None,  # or string if stale
    "temp_f": 83.6,        # tool-specific data
    "condition": "Clear",
    ...
}

# Error
{
    "ok": False,
    "error": "Weather unavailable",
    "error_code": "api_failed",
    "stale_notice": None
}
```

**Subclass pattern:**
```python
class SearchTools(BaseTool):
    def get_weather(self) -> dict:
        raw = get_current_weather()

        if raw.get("_live_failed"):
            return self.error("Weather unavailable", code="api_failed")

        return self.success({
            "temp_f": raw.get("temp_f"),
            "condition": raw.get("condition"),
        }, stale_result=raw)

# Module-level instance
_search = SearchTools()

# Backwards compat function
def get_weather() -> dict:
    return _search.get_weather()
```

## Rationale
**BaseAPIHandler:**
- Centralizes cache pattern — all API files use `self.call()` instead of direct `cache` imports
- Prevents cache collisions — `CACHE_PREFIX` validation ensures namespacing
- Enforces credential loading pattern — `_get_client()` abstract method
- Backwards compatible — module-level functions preserve existing imports

**BaseTool:**
- Enforces consistent return shape — agent always gets `ok`, `stale_notice`, and data
- Automatic stale_notice extraction — pass `stale_result` to `success()` and it handles the metadata
- Internal key isolation — strips `_stale`, `_cached_at` from returned data
- Error handling standardization — all errors have `ok=False`, `error`, `error_code`

**Class over module:**
- Namespaced methods — `self.call()` vs generic `call()`
- Testable — can mock the class instance
- Extensible — future methods can be added without breaking imports
- Consistent with CacheManager pattern (Layer 7)

## Consequences
### Positive
- Adding a new API client: subclass `BaseAPIHandler`, set `CACHE_PREFIX`, implement `_get_client()`, write methods
- Adding a new tool: subclass `BaseTool`, call `success()` or `error()` in every function
- Agent always gets consistent shape with `ok` and `stale_notice` fields
- Backwards compatible via module-level functions
- Pattern captured in code, not documentation

### Negative
- Additional class hierarchy
- Module-level functions add boilerplate
- Requires migration of existing files

### Mitigations
- Migrate incrementally (one API file at a time)
- Keep module-level functions for backwards compat
- Add comprehensive test coverage
- Document migration pattern in ADR

## Implementation
**Files created:**
- `api/_handler.py` — BaseAPIHandler class
- `tools/_tool.py` — BaseTool class
- `tests/test_base_classes.py` — 9 tests for both classes

**Files migrated:**
- `api/weather_api.py` — uses `WeatherAPIHandler(BaseAPIHandler)`
- `tools/search_tools.py` — uses `SearchTools(BaseTool)` for `get_weather()`

**Database hardening (side fix):**
- `infra/db/manager.py` — DBManager class with retry logic for lock errors
- `infra/db/schema.py` — WAL mode enabled for concurrent access
- `infra/db/cache.py` — migrated to use `db.exec()` instead of `_exec()`

**Test results:**
- 157/157 tests passing
- BaseAPIHandler tests: 4 passed
- BaseTool tests: 5 passed
