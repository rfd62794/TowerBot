# ADR-038: Tasks vs Jobs Separation

## Status
Proposed

## Context
PrivyBot currently conflates two distinct concepts under the term "task":

1. **Robert's personal tasks** (Brush Teeth, Daily standup, Record Shorts) — these have due dates Robert cares about, need reminders, and are managed via Google Tasks
2. **PrivyBot's autonomous work** (Email triage, YouTube sweep, Hourly fact, Chain steps) — these have prompts, model roles, execution state, and retry logic Robert never touches directly

Current problems:
- Local `tasks` and `personal_tasks` tables duplicate Google Tasks functionality
- Overdue notification spam: Jobs (autonomous work) leak into Task alert paths
- No clear boundary between user-facing tasks and system jobs
- Google Tasks sync is bolted on rather than being the canonical source
- Overdue deduplication is broken because Jobs don't have overdue semantics

## Decision
Establish a clear conceptual and architectural boundary:

| Concept | Owner | Storage | Examples | Overdue Alerts |
|---|---|---|---|---|
| **Tasks** | Robert | Google Tasks API | Daily standup, Brush Teeth, Record Shorts | Yes (via Google Tasks API) |
| **Jobs** | PrivyBot | Local SQLite job_queue | Email triage, YouTube sweep, Hourly fact, Chain steps | No (run/fail/retry, not overdue) |

### Core Principles

1. **Tasks = Google Tasks only**
   - No local `tasks` or `personal_tasks` tables
   - All task CRUD goes through Google Tasks API
   - Google Tasks is the single source of truth
   - Overdue alerts queried via Google Tasks API (`status=needsAction` and `due < today`)

2. **Jobs = local SQLite job_queue**
   - Table name remains `task_queue` (conceptually `job_queue`)
   - `queue_task()` conceptually `queue_job()`
   - Jobs have prompts, model roles, execution state
   - Jobs do not have due dates or overdue semantics
   - Jobs either run successfully, fail (logged), or retry
   - Template scheduler fires Jobs
   - Self-direction loop queues Jobs

3. **Sync boundary**
   - Google Tasks tools (`list_google_tasks`, `create_google_task`, etc.) are the only interface
   - `sync_google_tasks()` is the bridge between Google Tasks and local state
   - No direct local table access to task data

### Implementation Changes

1. **Remove local task tables**
   - Deprecate `tasks` table (used for weekly plan tasks)
   - Deprecate `personal_tasks` table
   - Migrate any critical data to Google Tasks or memory
   - Remove Check 5 (autonomous overdue) from `bot/scheduler.py` — Jobs don't send overdue notifications

2. **Google Tasks as canonical**
   - All task operations go through Google Tasks API tools
   - Overdue alerts query Google Tasks API directly
   - Check 9 in `bot/scheduler.py` becomes Google Tasks overdue query

3. **Job queue clarity**
   - Keep `task_queue` table name (backward compatible)
   - Document as "job_queue" conceptually
   - Ensure no overdue logic applies to jobs
   - Jobs have retry/fail semantics, not overdue semantics

### Overdue Deduplication Fix

With this separation, the overdue deduplication problem simplifies:

- **Tasks (Google Tasks):** Overdue alerts queried via Google Tasks API with proper dedup (API handles state)
- **Jobs (local):** No overdue alerts — they run/fail/retry, no spam

The current spam was Jobs leaking into Task alert paths. Separation eliminates this entirely.

## Implementation Plan

1. **Phase 1: Deprecation**
   - Add deprecation warnings to `tasks` and `personal_tasks` tables
   - Document migration path to Google Tasks
   - Stop rule on new features using local task tables

2. **Phase 2: Notification cleanup**
   - Remove Check 5 (autonomous overdue) from `bot/scheduler.py`
   - Replace Check 9 (personal tasks overdue) with Google Tasks API query
   - Verify overdue spam stops

3. **Phase 3: Migration**
   - Migrate weekly plan tasks to Google Tasks or memory
   - Migrate personal tasks to Google Tasks via sync
   - Drop `tasks` and `personal_tasks` tables from schema
   - Update all tool references to use Google Tasks API

4. **Phase 4: Documentation**
   - Update ADR to reflect completed migration
   - Document job queue patterns
   - Update tool descriptions to clarify Tasks vs Jobs

## Consequences

**Positive:**
- Clear conceptual boundary between user tasks and system jobs
- Overdue deduplication fixed by architectural separation
- Google Tasks as single source of truth eliminates sync drift
- Simpler notification logic (no cross-path leakage)
- Reduced schema complexity (fewer tables)

**Negative:**
- Migration effort for existing local task data
- Google Tasks API dependency (requires auth)
- Network latency for task operations (vs local DB)
- Job queue table name mismatch (`task_queue` vs conceptually `job_queue`)

**Risks:**
- Google Tasks API rate limits or outages
- Migration data loss if not handled carefully
- Breaking changes to tools using local task tables
- User confusion during transition period

## Alternatives Considered

- **Keep local task tables with dedup columns** — rejected: Doesn't solve conceptual confusion, adds schema complexity
- **Rename task_queue to job_queue** — rejected: Breaking change for minimal benefit, table name is internal
- **Add overdue flag to jobs** — rejected: Jobs don't have overdue semantics, this is the wrong abstraction
- **Hybrid approach (local + Google)** — rejected: Two sources of truth guarantees sync drift

## Notes

- Stop rule on new features using `tasks` or `personal_tasks` tables
- Google Tasks API quota: 1 billion requests/day per project (not a concern)
- Job queue table name kept for backward compatibility, documented as conceptually `job_queue`
- This ADR supersedes any prior ADR that assumed local task tables as canonical
