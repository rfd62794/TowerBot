# ADR-024: fetch_url Browser Tool

## Status
Accepted

## Context
DuckDuckGo search returns only snippets — the agent had no way to read full page content.

**Problem:**
- Search results provide only title, URL, and brief snippet
- Agent cannot read full articles or detailed content
- Grounding incomplete without full page access
- User asks "what does that article say" but agent only has snippet

**Existing Options:**
- web_search: returns snippets only
- wiki_lookup: Wikipedia-specific, not general web pages
- No tool for reading arbitrary web pages

## Decision
Create `fetch_url` tool using FetchAPIHandler(BaseAPIHandler) pattern.

### Architecture
```python
# api/fetch_api.py

class FetchAPIHandler(BaseAPIHandler):
    CACHE_PREFIX = "fetch"
    
    HEADERS = {
        "User-Agent": "PrivyBot/1.0 (personal tool)",
        "Accept": "text/html,application/xhtml+xml",
        "Accept-Language": "en-US,en;q=0.9",
    }
    
    def _get_client(self):
        return None  # requests needs no client
    
    def fetch_url(self, url: str, max_chars: int = 3000) -> dict:
        params_hash = self.hash(url, max_chars)
        
        def _live() -> dict:
            from bs4 import BeautifulSoup
            
            response = requests.get(
                url,
                headers=self.HEADERS,
                timeout=10,
                allow_redirects=True
            )
            response.raise_for_status()
            
            soup = BeautifulSoup(response.text, "html.parser")
            
            # Remove noise elements
            for tag in soup(["script", "style", "nav", "header", "footer", "aside", "form"]):
                tag.decompose()
            
            # Extract title
            title = ""
            if soup.title and soup.title.string:
                title = soup.title.string.strip()
            
            # Extract text
            text = soup.get_text(separator=" ", strip=True)
            text = re.sub(r'\s+', ' ', text).strip()
            
            return {
                "url": url,
                "title": title,
                "content": text[:max_chars],
                "char_count": len(text),
                "truncated": len(text) > max_chars,
                "status_code": response.status_code
            }
        
        return self.call("page", params_hash, _live, stale_ok=True)

# Module-level instance
fetch_api = FetchAPIHandler()
```

### Tool Layer Integration
```python
# tools/search_tools.py

class SearchTools(BaseTool):
    def fetch_url(self, url: str, max_chars: int = 3000) -> dict:
        from tools.api.fetch_api import fetch_api
        
        raw = fetch_api.fetch_url(url, max_chars)
        
        if raw.get("_live_failed"):
            return self.error(f"Could not fetch: {url}")
        
        if "error" in raw:
            return self.error(raw["error"])
        
        return self.success({
            "url": url,
            "title": raw.get("title", ""),
            "content": raw.get("content", ""),
            "char_count": raw.get("char_count", 0),
            "truncated": raw.get("truncated", False)
        }, stale_result=raw)
```

### Cache Strategy
- TTL: 3600 seconds (1 hour per URL)
- Cache key: hash(url, max_chars) — includes max_chars to distinguish different truncation levels
- stale_ok=True — stale page content better than no page
- Reduces repeated fetches of same page

### Content Extraction
- Removes noise elements: script, style, nav, header, footer, aside, form
- Extracts title from page title tag
- Extracts text content with BeautifulSoup.get_text()
- Normalizes whitespace with regex: `re.sub(r'\s+', ' ', text).strip()`

### Truncation Policy
- Default max_chars: 3000
- User can override up to 5000 characters
- Truncated flag indicates more content exists

## Rationale
**Why requests not httpx:**
- weather_api.py uses requests
- Consistency within codebase
- Both work — consistency wins
- No new dependency needed

**Why BeautifulSoup:**
- Robust HTML parsing
- Handles malformed HTML gracefully
- Easy element removal (noise tags)
- Standard library for web scraping

**Why cache per URL+max_chars:**
- fetch_url("example.com", 1000) and fetch_url("example.com", 5000) are different results
- Caching the 1000-char version and returning it for 5000-char request is a bug
- Default max_chars=3000 means most calls share the same cache entry naturally

**Why stale_ok=True:**
- Stale page content better than no page
- Web pages change slowly
- 1-hour TTL is reasonable for most content

## Consequences
### Positive
- Full page content available to agent
- web_search → fetch_url chain enabled
- Stale page content served on failure
- 1-hour cache reduces repeated fetches
- Better grounding for factual questions

### Negative
- Additional dependency (beautifulsoup4)
- HTTP request latency on cache miss
- Large pages may take time to parse

### Mitigations
- Cache reduces repeated fetches
- 10-second timeout prevents hanging
- Noise removal reduces content size
- Truncation limits response size
