# ADR-026: RateLimitManager Layer

## Status
Planned (not yet built)

## Context
API rate limits are currently untracked in PrivyBot:
- 429 errors handled reactively in CacheManager
- No proactive rate limit awareness
- No quota tracking per service
- No visibility into API usage patterns
- No way to know when approaching limits

**Current State:**
- YouTube Analytics: 10k units/day quota
- YouTube Data: 10k units/day (100 per search)
- Gmail: 5 req/second
- Steam: 200 req/5min
- SteamSpy: ~4 req/minute (measured)
- ITAD: 1000 req/day
- DuckDuckGo: ~20 req/minute (estimated)
- Reddit: 60 req/minute
- Open-Meteo: 10k req/day
- OpenRouter: handled by model_manager

**Problem:**
- No visibility into quota consumption
- No proactive degradation before 429
- No usage analysis for optimization
- Rate limit violations cause tool failures
- No way to budget API calls strategically

## Decision
Create `core/rate_limits.py` — RateLimitManager as Layer 7b (parallel to CacheManager).

### Architecture
```python
# core/rate_limits.py

class RateLimitManager:
    """
    Single owner of API rate limit tracking.
    Tracks quota usage, logs all calls, provides visibility.
    """

    def __init__(self):
        self._limits = {
            "youtube_analytics": {"quota": 10000, "window": "day"},
            "youtube_data": {"quota": 10000, "window": "day"},
            "gmail": {"quota": 5, "window": "second"},
            "steam": {"quota": 200, "window": "5min"},
            "steamspy": {"quota": 4, "window": "minute"},
            "itad": {"quota": 1000, "window": "day"},
            "ddg": {"quota": 20, "window": "minute"},
            "reddit": {"quota": 60, "window": "minute"},
            "openmeteo": {"quota": 10000, "window": "day"},
        }

    def check_limit(self, service: str) -> tuple[bool, int]:
        """
        Check if service is within rate limit.
        Returns (allowed, remaining_quota).
        """

    def log_call(self, service: str, success: bool) -> None:
        """
        Log an API call for usage analysis.
        """

    def get_status(self) -> dict:
        """
        Return quota status for all services.
        For /status command.
        """

# Singleton
rate_limits = RateLimitManager()
```

### Database Tables
```sql
-- api_rate_limits
CREATE TABLE api_rate_limits (
    service TEXT PRIMARY KEY,
    quota INTEGER NOT NULL,
    window TEXT NOT NULL,  -- 'second', 'minute', '5min', 'day'
    used INTEGER DEFAULT 0,
    window_start TEXT NOT NULL,  -- ISO 8601 timestamp
    updated_at TEXT NOT NULL
);

-- api_call_log
CREATE TABLE api_call_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    service TEXT NOT NULL,
    endpoint TEXT,
    success BOOLEAN NOT NULL,
    error TEXT,
    duration_ms INTEGER,
    called_at TEXT NOT NULL
);
```

### BaseAPIHandler Integration
```python
# tools/api/_handler.py

class BaseAPIHandler:
    def call(self, suffix: str, params_hash: str, live_fn: Callable, stale_ok: bool) -> dict:
        # Check rate limit before live call
        from core.rate_limits import rate_limits
        allowed, remaining = rate_limits.check_limit(self.CACHE_PREFIX)
        
        if not allowed:
            return {"error": f"Rate limit exceeded for {self.CACHE_PREFIX}", "_live_failed": True}
        
        # Make live call
        start = time.time()
        try:
            result = live_fn()
            success = True
            error = None
        except Exception as e:
            success = False
            error = str(e)
            raise
        finally:
            duration_ms = int((time.time() - start) * 1000)
            rate_limits.log_call(self.CACHE_PREFIX, success, error, duration_ms)
        
        # Cache and return
        return self.cache.set(...)
```

### /status Command Update
```python
# core/router.py

def get_status() -> dict:
    from core.rate_limits import rate_limits
    quota_status = rate_limits.get_status()
    return {
        "model_status": model_manager.get_status_report(),
        "quota_status": quota_status,
        ...
    }
```

## Rationale
**Why RateLimitManager:**
- Proactive rate limit awareness — prevent 429s before they happen
- Quota visibility — know when approaching limits
- Usage analysis — optimize API call patterns
- Consistent with CacheManager pattern — both are singletons managing a resource
- Layer 7b placement — parallel to CacheManager, both sit above API layer

**Why DB tables:**
- Persistent quota tracking across restarts
- Call log for historical analysis
- Window-based tracking (sliding windows for time-based limits)

**Why BaseAPIHandler integration:**
- Centralized check point — all API calls go through BaseAPIHandler
- No changes needed per API file
- Consistent enforcement across all services

## Consequences
### Positive
- Proactive rate limit awareness
- Quota visibility in /status command
- Graceful degradation before 429
- Call logging for usage analysis
- Better API budgeting

### Negative
- Additional database writes on every API call
- More complex BaseAPIHandler
- Need to track time windows correctly

### Mitigations
- Batch call log writes (buffer and flush)
- Use efficient time window tracking
- Add comprehensive tests
- Document known limits clearly

## Implementation Order
1. Create DB tables (api_rate_limits, api_call_log)
2. Implement RateLimitManager class
3. Add check_limit() and log_call() methods
4. Integrate into BaseAPIHandler.call()
5. Update /status command to show quota status
6. Add tests for rate limit enforcement
