# ADR-008: Telegram as Primary Interface

## Status
Accepted

## Context
PrivyBot needs a user interface for interaction. Options considered:
- **Web UI**: Browser-based interface
- **CLI**: Command-line interface
- **Telegram Bot**: Chat-based interface via Telegram Bot API

Each has trade-offs in terms of:
- Accessibility (mobile, desktop)
- Real-time notifications
- Setup complexity
- User experience

## Decision
Use Telegram Bot API as the primary interface.

### Why Telegram Over Web UI
- **Mobile-first**: Telegram is available on mobile, desktop, web
- **Push notifications**: Real-time alerts without polling
- **No server hosting**: Telegram handles hosting, no need for web server
- **Familiar UX**: Chat interface is intuitive
- **Rich media**: Support for images, files, formatting

**Rationale:** Web UI requires hosting, SSL, and maintenance. Telegram provides all infrastructure. Chat interface is natural for AI assistant.

### Why Polling Over Webhook
- **Simpler setup**: Polling requires no public URL or SSL
- **No server requirements**: Webhook needs public endpoint
- **Reliable for single-user**: Polling is sufficient for one user
- **Easier local development**: No tunneling (ngrok) required

**Rationale:** Webhooks are better for scale, but polling is simpler for single-user bot. No need for public URL or SSL certificate.

### Why Single-User Gate (ALLOWED_CHAT_ID)
- **Privacy**: Only authorized user can interact
- **Security**: Prevents unauthorized access
- **Cost control**: Prevents abuse of API quotas
- **Personal assistant focus**: Designed for one user, not multi-tenant

**Rationale:** PrivyBot is a personal assistant, not a public service. Single-user gate ensures privacy and security.

### Why Report Layer Is Decoupled from PTB
- **Report layer** (`report.py`) has no direct PTB imports
- Send function is injected at startup via `report.init_report()`
- Report layer is pure event logging — no Telegram knowledge
- Enables testing without PTB dependency

**Rationale:** Decoupling allows report layer to be tested independently. Send function injection enables flexibility (could swap Telegram for another interface).

## Consequences
### Positive
- No server hosting required
- Mobile-friendly with push notifications
- Simple setup (no SSL, no public URL)
- Familiar chat interface
- Secure single-user access

### Negative
- Dependency on Telegram service
- Polling has slight latency vs webhook
- Limited to Telegram platform
- Requires Telegram account

### Mitigations
- Polling is sufficient for single-user scale
- Telegram is reliable and widely used
- Report layer decoupling enables future interface swaps
- Single-user gate ensures security

## Alternatives Considered
**Web UI:** Rejected due to hosting complexity and lack of push notifications.

**CLI:** Rejected due to poor mobile experience and no push notifications.

**Webhook:** Rejected due to setup complexity (public URL, SSL) for single-user use case.

**Multi-user:** Rejected due to privacy concerns and personal assistant focus.
