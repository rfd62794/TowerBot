# ADR-011: Tool Architecture

## Status
Accepted

## Context
PrivyBot needs to extend the AI agent's capabilities beyond pure conversation. The agent needs to:
- Access external data (YouTube, Steam, IsThereAnyDeal)
- Persist information (memory management)
- Perform actions (thread naming, memory operations)

Options considered:
- **Tools as layers**: Add tool capabilities as layers in the architecture
- **Hardcoded agent logic**: Embed tool logic directly in agent.py
- **Plug-in pattern**: Tools as plug-ins registered via TOOL_REGISTRY

## Decision
Use plug-in pattern with TOOL_REGISTRY for tool architecture.

### Why Tools Are Plug-Ins Not Layers
- **Layers are structural**: Layers (transport, router, agent, report, memory, db) define the application architecture
- **Tools are capabilities**: Tools extend agent capabilities without changing architecture
- **Dynamic discovery**: Tools can be added/removed without touching layer structure
- **Separation of concerns**: Layers handle data flow, tools handle domain-specific actions

**Rationale:** Adding tools as layers would blur architectural boundaries. Plug-ins keep layers focused on structural concerns while tools provide domain capabilities.

### Why TOOL_REGISTRY Pattern
- **Centralized discovery**: Single source of truth for available tools
- **OpenAI compatibility**: Tool definitions match OpenAI function calling schema
- **Type safety**: Function references (not strings) checked at import time
- **Easy to audit**: All tools visible in one file
- **Simple to add/remove**: Single edit per tool

**Rationale:** TOOL_REGISTRY provides a clean interface between agent and tools. Agent discovers tools by reading registry, no hardcoded tool knowledge.

### How Tools Connect to Agent
1. **Tool definition**: Schema defined in TOOL_REGISTRY (OpenAI function format)
2. **Tool implementation**: Function in memory.py (or tools/ after refactoring)
3. **Agent discovery**: Agent reads TOOL_REGISTRY to get available tools
4. **OpenRouter integration**: Agent sends tool definitions to OpenRouter
5. **Tool execution**: OpenRouter returns tool_calls, agent executes via `_execute()`
6. **Result feedback**: Tool results fed back to OpenRouter for completion

**Rationale:** This pattern enables OpenRouter to decide when to call tools. Agent acts as executor, not decision-maker.

### Tool Philosophy
- **Agent-driven**: Agent decides when to call tools, not the user
- **Transparent**: All tool operations are logged via report layer
- **Idempotent**: Tools should be safe to call multiple times
- **Error-safe**: Tools never raise exceptions, always return error dict
- **Caching**: Tools cache results where appropriate (per ADR-003)

**Rationale:** Agent-driven tools enable autonomous operation. Transparency builds trust. Idempotency and error-safety ensure reliability.

## Consequences
### Positive
- Tools can be added without architectural changes
- Agent autonomously decides tool usage
- Centralized tool discovery and management
- OpenAI-compatible function calling
- Clear separation between architecture and capabilities

### Negative
- TOOL_REGISTRY becomes single file bottleneck
- Manual registration required for new tools
- Tool logic scattered across files (memory.py, tools/)
- Agent has no tool-specific knowledge (must be generic)

### Mitigations
- Keep registry organized with comments
- Consider splitting registry by domain if it grows
- Refactor to api/ and tools/tools/ for better organization (per ADR-001)
- Agent genericity is intentional — tools encapsulate domain knowledge

## Alternatives Considered
**Tools as layers:** Rejected due to blurring architectural boundaries.

**Hardcoded agent logic:** Rejected due to lack of flexibility and maintainability.

**String-based function resolution:** Rejected due to lack of type safety.

**User-driven tool calls:** Rejected due to reducing agent autonomy.
