# ADR-023: DBManager

## Status
Accepted

## Context
Database access in `infra/db/` is scattered across multiple files:
- `infra/db/cache.py` — direct `_exec()` calls
- `infra/db/schema.py` — direct `_exec()` calls
- `infra/db/__init__.py` — direct `_exec()` calls
- `infra/db/*.py` — 12+ files with direct `_exec()` calls

Problems:
- No centralized database access management
- No retry logic for lock errors
- SQLite default journal mode causes lock contention under concurrent access
- Each `_exec()` call opens/closes connection rapidly
- Test failures due to "database is locked" errors
- No connection pooling strategy
- WAL mode alone insufficient for rapid concurrent writes

**Initial attempt:**
- Enabled WAL mode in `init_db()` with `PRAGMA journal_mode=WAL`
- Result: Lock errors persisted in BaseAPIHandler tests
- Conclusion: WAL mode helps but retry logic needed

## Decision
Create `infra/db/manager.py` — DBManager class as the single owner of all database access.

### Architecture
```python
# infra/db/manager.py

class DBManager:
    """
    Single owner of all database access.
    Wraps _exec() with automatic retry on lock errors.
    All database access should go through db.exec().
    """

    def __init__(self, max_retries: int = 5, base_backoff: float = 0.1):
        self.max_retries = max_retries
        self.base_backoff = base_backoff

    def exec(self, sql: str, params: Tuple = (), commit: bool = False) -> Cursor:
        """
        Execute SQL with automatic retry on lock errors.
        Exponential backoff: 0.1s, 0.2s, 0.4s, 0.8s, 1.6s
        """
        for attempt in range(self.max_retries):
            try:
                return _exec(sql, params, commit)
            except OperationalError as e:
                if "locked" in str(e).lower() and attempt < self.max_retries - 1:
                    backoff = self.base_backoff * (2 ** attempt)
                    time.sleep(backoff)
                    continue
                raise

    def exec_many(self, sql: str, params_list: list[Tuple], commit: bool = False) -> None:
        """Execute SQL with multiple parameter sets."""
        ...

    def get_connection(self) -> sqlite3.Connection:
        """Get the current database connection."""
        ...

# Singleton
db = DBManager()
```

### Migration Pattern
```python
# BEFORE
from core.db.schema import _exec
_exec("INSERT INTO tool_cache ...", params, commit=True)

# AFTER
from core.db.manager import db
db.exec("INSERT INTO tool_cache ...", params, commit=True)
```

### WAL Mode (Retained)
```python
# infra/db/schema.py — init_db()
_conn.execute("PRAGMA journal_mode=WAL")
_conn.execute("PRAGMA synchronous=NORMAL")
```
WAL mode remains enabled as it provides better concurrency, but DBManager handles the remaining lock contention.

## Rationale
**Why DBManager:**
- Centralized retry logic — exponential backoff handles transient locks
- Single owner of database access — consistent pattern across all files
- Future extensibility — connection pooling, query logging, metrics can be added without changing callers
- Test reliability — automatic retry makes tests less flaky under concurrent access
- Consistent with CacheManager pattern (Layer 7) — both are singletons managing a resource

**Why not just WAL mode:**
- WAL mode alone didn't fix the lock issue in BaseAPIHandler tests
- Rapid consecutive writes still cause contention even with WAL
- Retry logic is needed for production reliability

**Why exponential backoff:**
- Simple and effective for transient locks
- Progressive delay gives other processes time to complete
- 5 retries with 1.6s max wait balances responsiveness and reliability

## Consequences
### Positive
- Lock errors eliminated in tests (157/157 passing)
- Centralized retry policy — consistent across all database access
- Future connection pooling possible without changing callers
- Query logging and metrics can be added at one location
- Consistent with CacheManager architectural pattern

### Negative
- Additional layer of indirection
- All database files need migration to `db.exec()`
- Partial migration (only cache.py done so far)

### Mitigations
- Migrate incrementally per file
- Keep `_exec()` available for transition period
- Add comprehensive test coverage
- Document migration pattern in ADR
