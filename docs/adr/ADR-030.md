# ADR-030: Deduplication Strategy

**Status:** Accepted

**Context:**
Three sources were generating duplicate rows across the live DB:

1. *Personal tasks* — tests creating tasks without reliable teardown. Morning briefing showed test artifacts ("Snooze me" ×4, "YouTube analytics review" ×6). 341 leaked rows cleaned manually before this ADR.
2. *Commitments* — `save_commitment()` fires on trigger phrase detection. Repeated phrasing across sessions produced 167 duplicate rows.
3. *History tables* — `record_channel_day()` and `record_weather_day()` used plain INSERT. Briefing firing twice in one day created duplicate date rows.

**Decision:**
Dedup applied at the appropriate layer for each table:

1. `personal_tasks`: UNIQUE partial index on `(title, due_date) WHERE status='pending'`. INSERT OR IGNORE on `add_personal_task()`. Partial scope is essential — recurring tasks legitimately repeat the same title across different dates; completed tasks of the same title are valid history. Only pending tasks with the same title on the same date are true duplicates.

2. `commitments`: Application-level 24h window check in `save_commitment()`. Exact text match within window returns `{"duplicate": True}` without inserting. Required adding `created_at` column (migration 8eabd8d). Window chosen over structural uniqueness because the same commitment legitimately recurs across weeks with fresh intent.

3. `channel_history`, `weather_history`: INSERT OR REPLACE — idempotent by date. No window logic needed; latest value for a date is always correct.

**Rationale:**
DB-level constraint for `personal_tasks` because the uniqueness condition is structural and the DB can enforce it without application context. Application-level for `commitments` because temporal context is required. INSERT OR REPLACE for history tables because they are append-by-date logs where recency wins.

**Consequences:**

*Positive:*
- Clean personal task list in morning briefing
- Commitment log reflects unique obligations, not repeated detections
- History tables are idempotent — briefing can fire multiple times safely

*Negative:*
- INSERT OR IGNORE is silent on collision — edge cases hard to debug without logging
- 24h window on commitments may miss near-duplicate text that represents the same intent

*Mitigations:*
- Debug log on INSERT OR IGNORE collision
- Fuzzy matching for commitments deferred to backlog — no evidence it's needed yet

**Implementation:**
- `infra/db/schema.py` — UNIQUE partial index
- `infra/db/personal_tasks.py` — INSERT OR IGNORE
- `infra/db/goals.py` — 24h window check in `save_commitment()` 
- `infra/db/history.py` — INSERT OR REPLACE
- Commits: 8eabd8d (migration), 75e508c (test fix), 594a6de (index + logic)
- Tests: 204/204 passing
