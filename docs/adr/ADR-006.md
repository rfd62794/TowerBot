# ADR-006: Autonomous Memory Management

## Status
Accepted

## Context
PrivyBot needs to remember information across conversations to be useful as a personal assistant. Initial approaches considered:
- Chat-based bootstrap (user provides context in chat)
- Manual database insertion
- File-based configuration

These approaches have problems:
- Chat-based bootstrap is tedious and error-prone
- Manual database insertion requires technical knowledge
- File-based configuration is hard to update dynamically

### The EIC Hallucination Incident
- owl-alpha model hallucinated "Electronic Intelligence Community" as a Roblox game
- Cold-start agent with no memory invented false information
- This demonstrated the need for seeded memory to prevent hallucinations
- YAML seed (context.yaml) was created to provide accurate initial context

## Decision
Autonomous memory management via OpenRouter function calling tools.

### Why Agent Manages Its Own Memory
- The agent can decide what to remember based on conversation context
- Tools (save_memory, update_memory, retire_memory, get_memories) give the agent control
- No manual intervention required for most memory operations
- Agent can correct its own memory when it learns new information

### Why Transparency Is a Hard Rule
- Every memory operation is logged via report layer
- User can view all memories via `/memories` command
- Memory operations are visible in Telegram notifications
- No hidden memory writes — user always knows what's being stored
- **Origin**: Design decision after EIC hallucination — user must see what agent believes to catch errors early

**Rationale:** Trust requires transparency. If the agent silently modifies memory, users lose trust. All memory operations must be visible. Transparency enables user to catch hallucinations and corrections early.

### Why Five Layers (Technical/Project/Personal/Business/Content)
- **Technical**: Codebase decisions, architecture, debugging notes
- **Project**: Project-specific context, goals, status
- **Personal**: User preferences, habits, personal context
- **Business**: Business decisions, contacts, financial context
- **Content**: Content ideas, creative work, drafts

**Rationale:** Different types of information need different organization. Five layers provide structure without being overly rigid.

### Why Corrections Are Always Accepted
- When the agent calls `update_memory`, the update is applied immediately
- No confirmation required for memory corrections
- Agent can fix its own mistakes without user intervention
- Reduces friction in memory maintenance

**Rationale:** The agent should be able to self-correct. Requiring confirmation for every memory update would make the system cumbersome.

### Why YAML Seeding Over Chat Bootstrap
- `context.yaml` contains 26 seed memories
- `seed.py` loads YAML and inserts into SQLite (INSERT OR REPLACE, idempotent)
- Seeding happens once at setup, not during conversation
- Chat is for incremental updates, not bulk initialization
- **Purpose**: Prevent cold-start hallucination like EIC incident — agent starts with accurate context

**Rationale:** Bulk memory initialization via chat is tedious. YAML seeding provides a clean setup process. Idempotent INSERT OR REPLACE allows re-running seed.py without duplicates. Seeded memory prevents hallucinations from cold-start agent.

## Consequences
### Positive
- Agent can autonomously build and maintain memory
- Transparency builds user trust
- Structured memory layers improve organization
- Easy setup via YAML seeding
- Agent can self-correct without friction

### Negative
- Agent may store incorrect information (corrections required)
- Memory can grow unbounded without cleanup
- YAML seeding requires manual updates for new seed data
- Transparency adds notification overhead

### Mitigations
- `/memories` command allows user to review and correct memory
- `retire_memory` tool allows agent to remove outdated information
- Periodic memory review recommended
- YAML seeding is idempotent — safe to re-run

## Alternatives Considered
**Chat-based bootstrap:** Rejected due to tediousness and error-proneness.

**Manual database insertion:** Rejected due to requiring technical knowledge.

**File-based configuration:** Rejected due to lack of dynamic updates.

**No transparency:** Rejected due to trust concerns.
