# ADR-004: Layered Architecture

## Status
Accepted

## Context
PrivyBot was initially implemented as a monolith (`legacy_privybot.py`) with all functionality in a single file. This made the codebase difficult to:
- Test individual components in isolation
- Replace implementations (e.g., swap Telegram for Discord)
- Understand data flow
- Maintain as the codebase grew

The monolith mixed Telegram Bot API handling, command parsing, AI agent logic, memory management, and database operations in one place.

## Decision
Refactor into six distinct layers with strict import boundaries:

### Layer Structure
```
Layer 0: Transport (transport.py)  — Telegram Bot API integration
Layer 1: Router (router.py)         — Command parsing, delegation
Layer 2: Agent (agent.py)           — OpenRouter calls, tool execution
Layer 3: Report (report.py)         — Event logging, notifications
Layer 4: Memory (memory.py)         — Tool definitions, implementations
Layer 5: Database (db.py)           — SQLite persistence
```

### Import Rule
- **Layer N** can import from **Layer N+1** (lower layers)
- **Layer N** cannot import from **Layer N-1** (higher layers)
- No circular imports
- Each layer owns its domain completely

### Layer Responsibilities

**Layer 0: Transport**
- Owns Telegram Bot API (PTB) integration
- Handles incoming messages
- Sends replies with retry logic
- No business logic — only transport

**Layer 1: Router**
- Parses commands (`/think`, `/claude`, `/new`, `/memories`, `/models`, `/status`)
- Manages thread-per-chat mapping
- Delegates to agent for AI responses
- No Telegram knowledge

**Layer 2: Agent**
- Calls OpenRouter API
- Handles tool execution
- Injects memory context
- Manages model rotation on 429 errors
- No Telegram or command knowledge

**Layer 3: Report**
- Logs key bot events
- Sends Telegram notifications
- Pure event layer — injected send function at startup
- No business logic

**Layer 4: Memory**
- Defines tool definitions for OpenRouter
- Implements tool functions
- Calls database functions
- No knowledge of Telegram, routing, or agent

**Layer 5: Database**
- SQLite operations
- Schema definition
- Thread-safe connection management
- Pure database — no business logic

### Why This Layer Order
The order reflects data flow from external interface to persistence:
1. **Transport** receives external input (Telegram)
2. **Router** parses and routes commands
3. **Agent** processes with AI and tools
4. **Report** logs events asynchronously
5. **Memory** provides tool capabilities
6. **Database** persists all state

## Consequences
### Positive
- Clear separation of concerns — each layer has one job
- Testability — layers can be tested in isolation
- Replaceability — swap a layer without affecting others
- Understandability — data flow is explicit
- No circular dependencies — strict top-down imports

### Negative
- More files to manage
- Initial refactoring effort from monolith
- Slight overhead of function calls across layers

### Mitigations
- Keep layers focused on single responsibilities
- Document import rules in ARCHITECTURE.md
- Use clear naming conventions
- Maintain layer boundaries strictly

## Alternatives Considered
**Monolith:** Keep everything in one file. Rejected due to maintainability issues as codebase grew.

**Two-layer (UI + Logic):** Split into interface and logic only. Rejected because logic itself was still mixed (Telegram + AI + DB).

**Microservices:** Split into separate services. Rejected due to complexity for a single-user bot — SQLite and in-memory are sufficient.
