# ADR-029: Three-Tier Tool Architecture

## Status
Vision (not yet planned)

## Context

### Current State
- All tools are "system tools" — built by the developer, hardcoded, always available
- `tools/registry.py` is the canonical source for tool definitions
- ADR-028 defines `@tool` decorator with `category` and `source` metadata
- No mechanism for user-built tools or agent-discovered tools
- No approval workflow for experimental tools
- No sandboxing for untrusted code

### Problems
1. **Single tier only:** All tools are treated equally — no distinction between trusted system tools and experimental tools
2. **No user contribution pathway:** Users cannot build and promote their own tools
3. **No autonomous discovery:** Agent cannot discover new APIs and build tools overnight
4. **No safety boundaries:** Experimental tools have same privileges as system tools
5. **No approval workflow:** No mechanism to review and promote tools from experimental to trusted

## Decision

### Solution
Implement a three-tier tool architecture with distinct trust levels, promotion pathways, and sandboxing requirements.

### Architecture
```
System Tools     ← current, always available
                   built by developer, hardcoded
                   tier="system"
                   approved=True (implicit)

User Tools       ← user-built or promoted from agent
                   built intentionally, refined
                   tier="user"
                   approved=True
                   promoted_at="2026-06-01"

Agent Tools      ← autonomous, approval-gated
                   discovered overnight, experimental
                   tier="agent"
                   approved=False (pending review)
                   sandboxed=True
```

### Promotion Pathway

#### Agent → User
```
Agent discovers new API overnight:
  1. Reads documentation
  2. Writes tool definition
  3. Writes test
  4. Runs test in sandbox
  5. If test passes → saves to tools/agent/ as pending

Morning review:
  /review agent_tools
  → Shows pending tools with code and test results

User approval:
  /approve get_crypto_price
  → Moves to tools/user/
  → Sets tier="user"
  → Sets approved=True
  → Sets promoted_at=now
  → Bot restarts with new tool

User rejection:
  /reject get_crypto_price
  → Deletes from tools/agent/
  → Logs rejection reason
```

#### User → System
```
User tool becomes trusted over time:
  1. User tool runs successfully for N days
  2. No errors, no rate limit issues
  3. User requests promotion via PR
  4. Developer reviews code
  5. If approved → moves to tools/system/
  6. Sets tier="system"
  7. Becomes part of core bot
```

#### Agent Tool Constraints
- **Database:** Read-only access to production, write to isolated `agent_test.db`
- **Rate limits:** Default 1 call/minute, configurable per tool
- **Network:** Whitelist only approved domains (coingecko.com, etc.)
- **Execution:** 30-second timeout, 100MB memory limit
- **Persistence:** Cannot modify system tools or user tools
- **Side effects:** Cannot send messages, cannot modify production state

```
     Status: pending
     Test: FAILED (1/3)
     Code: [show tool definition]

```

#### Approve Command
```
/approve get_crypto_price

Actions:
  1. Move tools/agent/get_crypto_price.py → tools/user/get_crypto_price.py
  2. Update decorator: tier="user", approved=True, promoted_at=now
  3. Add to tools/user/__init__.py
  4. Restart bot
  5. Log promotion to deployments table
```

#### Reject Command
```
/reject get_crypto_price "API key required, not approved"

Actions:
  1. Delete tools/agent/get_crypto_price.py
  2. Log rejection with reason
  3. Notify agent to avoid rediscovery
```

## Implementation Timing

### Phase 1: Foundation (ADR-028)
- Implement `@tool` decorator with `tier` and `approved` fields
- Add `tier="system"` to all existing tools
- Set `approved=True` for all existing tools
- No functional change — metadata only

### Phase 2: User Tools
- Create `tools/user/` directory structure
- Implement `/approve` and `/reject` commands
- Implement promotion workflow (agent → user)
- Add review UI via `/review agent_tools`
- No agent discovery yet — manual user tool creation

### Phase 3: Agent Discovery
- Implement overnight scheduler slot
- Build API discovery engine
- Implement sandbox execution context
- Generate tool definitions from docs
- Generate and run tests in sandbox
- Save pending tools to `tools/agent/`

### Phase 4: User → System Promotion
- Implement PR-based promotion workflow
- Add promotion criteria (N days success, no errors)
- Developer review process
- Move to `tools/system/` on approval

## Consequences

### Positive
- Clear trust boundaries between tool tiers
- Users can contribute tools without developer intervention
- Agent can autonomously discover new capabilities
- Experimental tools are safely sandboxed
- Approval workflow ensures code review before promotion
- Gradual trust building from agent → user → system

### Negative
- Significant complexity: sandbox, approval workflow, discovery engine
- Requires isolated database context for agent tools
- Requires domain whitelisting for network access
- API discovery is non-trivial (LLM accuracy, doc parsing)
- Overnight discovery requires scheduler slot and monitoring
- User tools still need manual review for system promotion

### Mitigations
- Phase 1 (metadata only) is low-risk
- Phase 2 (user tools) is manageable without agent discovery
- Phase 3 (agent discovery) is the major build — can defer
- Sandbox constraints prevent damage from bad agent tools
- Approval workflow ensures human review before promotion
- Discovery engine can start with curated API list (not full web search)
