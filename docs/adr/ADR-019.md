# ADR-019: Staleness Budget per API

## Status
Accepted

## Context
Not all data ages at the same rate. Some data is highly time-sensitive (email, calendar), while other data is relatively stable (YouTube analytics, Steam player counts). The offline-first cache strategy needs per-API staleness budgets to:
- Avoid serving misleading stale data for time-sensitive APIs
- Allow longer stale fallback for stable data
- Reduce API quota usage where appropriate
- Set appropriate TTL values per API

## Decision
Define staleness budgets per API based on data volatility and user tolerance for staleness.

### Staleness Budget Matrix

| API | Data Type | TTL | Stale Budget | Rationale |
|-----|-----------|-----|--------------|-----------|
| weather_api | Current weather | 1 hour | 2 hours | Weather changes frequently, but 2-hour-old data is still useful |
| youtube_api (Analytics) | Channel metrics | 6 hours | 24 hours | Analytics update daily, 24h stale acceptable |
| youtube_api (Data) | Video metadata | 24 hours | 7 days | Metadata stable, rarely changes |
| steam_api | Owned games | 24 hours | 7 days | Game library changes infrequently |
| steamspy_api | Player counts | 24 hours | 48 hours | Player counts trend slowly |
| itad_api | Sale prices | 1 hour | 6 hours | Sales change frequently |
| gmail_api | Inbox summary | 5 minutes | 30 minutes | Email is time-sensitive |
| google_calendar_api | Events | 15 minutes | 2 hours | Calendar events need freshness |
| google_tasks_api | Task lists | 5 minutes | 1 hour | Tasks change frequently |
| ddg_api | Web search | 1 hour | 6 hours | Search results change moderately |
| wikipedia_api | Article summary | 24 hours | 7 days | Wikipedia articles stable |
| reddit_api | Search results | 1 hour | 6 hours | Reddit content changes frequently |

### TTL vs Stale Budget
- **TTL**: Cache freshness threshold — data considered "fresh" if within TTL
- **Stale Budget**: Maximum acceptable staleness — serve stale data if within budget

Example for weather_api:
- TTL: 1 hour (fresh if < 1h old)
- Stale Budget: 2 hours (serve stale if < 2h old, error if > 2h)

### Implementation
```python
# api/_base.py
STALE_BUDGETS = {
    "weather_api": 7200,      # 2 hours
    "youtube_api": 86400,     # 24 hours
    "steam_api": 604800,      # 7 days
    "gmail_api": 1800,       # 30 minutes
    # ... etc
}

def cached_api_call(tool_name: str, params_hash: str, live_fn: Callable,
                   ttl_seconds: int, stale_ok: bool = True) -> dict:
    # ... existing logic ...

    # On API failure, check staleness budget
    if stale_ok:
        stale = get_stale_cached_result(tool_name, params_hash)
        if stale:
            age_minutes = stale.get("_age_minutes", 0)
            budget = STALE_BUDGETS.get(tool_name, ttl_seconds * 2)
            if age_minutes * 60 <= budget:
                return stale
            else:
                logger.warning(
                    f"[API] {tool_name} stale data exceeds budget "
                    f"({age_minutes}min > {budget/60}min)"
                )

    return {"error": str(e), "_live_failed": True, "_stale": False}
```

### Morning Briefing Stale Detection
Health check uses staleness budgets to flag problematic data:
```python
# scheduler.py health_check
status = get_preload_status()
for entry in status:
    age_minutes = entry["age_minutes"]
    budget = STALE_BUDGETS.get(entry["tool_name"], 3600)
    if age_minutes * 60 > budget:
        # Flag as stale in morning briefing
        stale_tools.append(entry["tool_name"])
```

## Consequences
### Positive
- Time-sensitive APIs (Gmail, Calendar) have short staleness budgets
- Stable APIs (YouTube, Steam) can serve stale data longer
- Reduced API quota usage for stable data
- Clear per-API expectations documented

### Negative
- More configuration to maintain
- Stale budgets need periodic review
- Edge cases where data volatility changes

### Mitigations
- Document staleness budgets in ADR
- Review budgets quarterly
- Add metrics to track stale data usage
- Allow environment-specific overrides if needed

## Examples

### Weather API (time-sensitive)
```python
# TTL: 1 hour, Stale Budget: 2 hours
result = cached_api_call("weather_api", "hash123", live_fn, 3600)

# If API fails and cache is 90 minutes old:
# → Serve stale data (within 2h budget)
# → Notice: "⚠️ Data from 1h 30m ago"

# If API fails and cache is 3 hours old:
# → Error (exceeds 2h budget)
# → Notice: "Weather data too stale (3h old), API unavailable"
```

### YouTube Analytics (stable)
```python
# TTL: 6 hours, Stale Budget: 24 hours
result = cached_api_call("youtube_api", "hash456", live_fn, 21600)

# If API fails and cache is 18 hours old:
# → Serve stale data (within 24h budget)
# → Notice: "⚠️ Data from 18h ago"

# If API fails and cache is 30 hours old:
# → Error (exceeds 24h budget)
# → Notice: "Channel metrics too stale (30h old), API unavailable"
```

### Gmail (highly time-sensitive)
```python
# TTL: 5 minutes, Stale Budget: 30 minutes
result = cached_api_call("gmail_api", "hash789", live_fn, 300)

# If API fails and cache is 20 minutes old:
# → Serve stale data (within 30m budget)

```
