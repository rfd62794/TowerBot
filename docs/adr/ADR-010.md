# ADR-010: Context Window Strategy

## Status
Accepted

## Context
PrivyBot needs to manage conversation context for the AI agent. Key considerations:
- LLMs have limited context windows (typically 4k-32k tokens)
- Memory persists across conversations (autonomous memory management)
- Users may want to start fresh conversations (/new command)
- Thread isolation is important for different topics

Options considered:
- **Unlimited context**: Send all messages ever
- **Fixed window**: Last N messages only
- **Sliding window**: Last N messages, memory persists
- **Topic-based**: Group by topic, not time

## Decision
Use 10-message sliding window with thread isolation and persistent memory.

### Why 10 Messages Sliding Window
- **Balance**: 10 messages provides sufficient context without hitting token limits
- **Recent focus**: Most relevant context is recent
- **Token efficiency**: 10 messages typically fits within 4k-8k token limits
- **Performance**: Smaller context = faster inference

**Rationale:** Unlimited context would hit token limits. Fixed window is too rigid. Sliding window balances context depth with token efficiency.

### Why Thread Isolation
- Each Telegram chat has its own thread in SQLite
- Threads are isolated — messages from one chat don't leak to another
- `/new` command creates a new thread for the same chat
- Thread names are agent-driven (via name_thread tool)

**Rationale:** Thread isolation prevents context contamination. Different conversations should not influence each other.

### Why Memory Persists Across /new
- Memory is stored in separate table (memory) from messages
- `/new` creates new thread (clears message history)
- Memory is NOT cleared by `/new` — persists across threads
- Agent can still access saved memory in new threads

**Rationale:** Memory is long-term knowledge, not conversation context. Clearing memory on `/new` would lose valuable information. Message history is temporary context, memory is permanent knowledge.

### Why Thread Naming Is Agent-Driven
- Agent calls `name_thread` tool to name threads
- Names reflect conversation topic (e.g., "Raccoin development", "YouTube analytics")
- Names are stored in threads table
- User can view thread names via `/status` command

**Rationale:** Agent understands conversation context better than automatic naming. Manual naming would be tedious.

## Consequences
### Positive
- Efficient token usage with sliding window
- Thread isolation prevents context contamination
- Memory persists across conversations
- Agent-driven thread names are meaningful
- Fresh start capability with `/new`

### Negative
- 10 messages may lose important older context
- Thread isolation prevents cross-thread context sharing
- Memory persistence may cause stale information
- Agent may name threads inconsistently

### Mitigations
- Memory provides long-term context beyond sliding window
- User can manually retire stale memory
- Agent can correct memory when it learns new information
- `/status` shows thread names for clarity

## Alternatives Considered
**Unlimited context:** Rejected due to token limits and performance.

**Fixed window (all messages):** Rejected due to token limits and lack of freshness.

**Topic-based grouping:** Rejected due to complexity and lack of clear topic boundaries.

**Memory cleared on /new:** Rejected due to loss of valuable long-term information.
