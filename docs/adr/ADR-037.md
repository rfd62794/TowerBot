# ADR-037: Chain System — Six Primitive Architecture

## Status
Accepted

## Context
PrivyBot tasks run and complete with no continuity. There is no concept of
being mid-chain, waiting for approval, or resuming from a prior state. A
chain system is needed that survives reboots, supports human approval steps,
enables reactive and proactive triggers, and allows the agent to self-expand
by creating reusable templates from observed patterns.

## Decision

Six primitives are adopted as the permanent foundation. No others.

**Template** — The blueprint. A reusable chain definition stored as YAML
(canonical, human-authored or agent-promoted) or SQLite row (experimental,
agent-created at runtime). A Template is never a running instance.

**Chain** — The running instance of a Template. Tracks execution state:
current step, status, payload reference, and a full append-only trace.
Survives reboots. Resumable from any step.

**Step** — The smallest unit of work. Types: llm_call, tool_call,
approval_wait, condition_check, spawn_chain, transform. Consumes a Payload,
produces a Payload. Steps do not have their own schema — input and output
are Payloads, already validated.

**Payload** — The data traveling through a chain. Always structured JSON,
never freeform text. Stamped with type and schema_version. Persisted to
SQLite — never held in context memory. Enriched at each step.

**Schema** — The contract defining what a Payload type contains. One schema
definition, many payload instances. schema_version enables forward
compatibility without breaking existing payloads.

**Trigger** — What starts or resumes a chain. Types: schedule, event,
condition, approval, completion. All trigger types share a uniform
evaluation interface: qualifier evaluates to true → fire. The completion
trigger type enables automatic chain-to-chain sequencing.

## Storage

Live chains: SQLite (chains, chain_steps, chain_payloads,
pattern_candidates, approval_listeners tables).

Archived chains: Parquet, partitioned by month (YYYY-MM/). Written once on
chain completion. Full denormalized record — all steps, all payloads.

Templates: YAML files in templates/canonical/ (trusted, version-controlled)
or SQLite rows (experimental, agent-created). Promotion pipeline moves
experimental → canonical after N successful uses.

## Consequences

**Locked:**
- Six primitives only. No seventh primitive without a superseding ADR.
- Payload data field is always JSON. Freeform text payloads are a bug.
- Steps do not have schemas. Only Payloads have schemas.
- Chains are append-only in the steps log. No step record is ever deleted.
- Templates in templates/canonical/ are never modified by the agent —
  only appended. Deletion requires human instruction.
- Promotion is append-only. Agent adds experimental templates; it never
  modifies or deletes canonical ones.

**Deferred:**
- Chain branching (parallel paths). Linear-with-routing only for now.
- Level 2+ self-expansion (tool composition, tool acquisition).
- Model role routing within steps (designed, not implemented).
- Parquet archive implementation (Phase 21+).
- Full promotion pipeline implementation (Phase 20+).
