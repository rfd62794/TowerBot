# ADR-015: Heartbeat and Proactive Scheduling

## Context
PrivyBot was purely reactive — it only responded when messaged or at two fixed times (7AM morning briefing, 11:59PM nightly summary). It had no mechanism to surface observations proactively based on changing conditions. Content gaps, game trends, and time-sensitive alerts would go unnoticed until the user asked.

## Decision
Implement a three-part proactive system:

### 1. Hourly Heartbeat
- Runs every 60 minutes via `heartbeat_check()`
- Checks conditions and queues observations
- Flushes pending queue based on time/priority routing

### 2. Task Queue
New SQLite table `task_queue`:
- Stores observations with priority and scheduled time
- Fields: task_name, message, priority, created_at, scheduled_for, sent
- Functions: queue_observation, get_pending_observations, mark_sent, flush_morning_queue

### 3. Time-Aware Routing
`should_send_now(priority)` function:
- Sleep hours (midnight-7AM): only critical priority
- Late night (10PM-midnight): critical and high priority
- Normal hours (7AM-10PM): all priorities

## Rationale

### Why hourly not per-minute?
- Balance between responsiveness and resource usage
- 60-minute interval catches most meaningful changes
- Prevents notification fatigue
- Reduces API quota usage

### Why sleep hours suppression?
- User sleeps midnight-7AM Eastern
- Non-critical alerts during sleep are disruptive
- Queue overnight alerts for morning briefing
- Only critical alerts (e.g., system failures) should interrupt

### Why task queue vs direct send?
- Time-aware routing requires deferral
- Morning briefing needs to batch overnight alerts
- Enables "send at specific time" scheduling
- Prevents lost observations during sleep hours

### Why priority routing model?
- Three levels: critical, high, normal
- Critical: system failures, urgent deadlines
- High: content gaps, significant trends
- Normal: minor trends, informational
- Enables intelligent alert filtering

### Why morning briefing flushes queue?
- Overnight alerts should be reviewed at start of day
- Prevents alert spam during morning routine
- Provides context for the day ahead
- Single digest instead of multiple notifications

## Implementation
- `core/db.py`: task_queue table + 4 CRUD functions
- `core/scheduler.py`: 
  - should_send_now() routing function
  - heartbeat_check() with 3 condition checks
  - Updated run_scheduler() with hourly heartbeat
  - Updated morning_briefing() to flush queue
- `scripts/verify.py`: 3 new tests (table, heartbeat, routing)

## Consequences
- PrivyBot now surfaces observations proactively
- Content gaps detected 3-7 days in advance
- Game player spikes (>20%) queued for briefing
- Sleep hours respected for non-critical alerts
- Morning briefing includes overnight alerts
- 30/30 tests pass
- Minimal resource overhead (1 check/minute)
