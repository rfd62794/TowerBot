# ADR-039: Shell Execution Security Model

## Status: Accepted

## Decision
Two-layer model for shell command execution:

**Layer 1 — Named command registry (primary):**
- LLM always calls `run_named_command(name)` to execute commands
- Commands are resolved from `NAMED_COMMANDS` dict only
- No runtime command construction or string concatenation
- Each named command has a fixed command string, working directory, and description
- Unknown command names return an error with the list of available commands

**Layer 2 — Filtered raw execution (secondary):**
- `execute_shell()` runs two filter stages before any subprocess execution
- Stage 1: Verb whitelist — only approved verbs (nssm, uv, git, python, pytest, etc.) are allowed
- Stage 2: Pattern blocklist — dangerous patterns (rm, del, format, &&, ||, ;, etc.) are blocked
- Both stages must pass before execution
- Blocked attempts are logged at WARNING level and never reach subprocess
- Timeout enforcement prevents hanging commands

## Consequences
- New shell operations require deliberate modification of the NAMED_COMMANDS registry
- Attack surface is bounded by the approved verb list and blocklist patterns
- The security model is permanent — future phases must use this same two-layer approach
- Named commands are preferred for operational use; filtered raw execution is for ad-hoc needs
- All shell execution is logged for audit trail
