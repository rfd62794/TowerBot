# ADR-007: SQLite as Persistence Layer

## Status
Accepted

## Context
PrivyBot needs persistence for:
- Conversation threads and messages
- Memory (autonomous memory management)
- Model throttle tracking
- Tool result caching
- Channel history (daily metrics)
- Key-value cache

Options considered:
- **Postgres**: Robust but requires external service, setup complexity
- **Redis**: Fast but requires external service, volatile without persistence
- **Files**: Simple but no querying, no ACID guarantees
- **SQLite**: Embedded, ACID-compliant, no external service

## Decision
Use SQLite as the single persistence layer.

### Why SQLite Over Postgres/Redis/Files
- **No external service**: SQLite is embedded in Python, no setup required
- **ACID compliance**: Transactions ensure data integrity
- **Portability**: Single file (privy.db) can be backed up, moved, versioned
- **Sufficient performance**: Single-user bot doesn't need distributed scale
- **SQL querying**: Rich query language for complex operations

**Rationale:** For a single-user bot, Postgres/Redis are overkill. Files lack ACID guarantees. SQLite provides the right balance of simplicity and robustness.

### Why All State in One privy.db
- Single file for all persistence needs
- Easy backup (copy one file)
- Easy migration (move one file)
- Atomic transactions across all tables
- No need to manage multiple databases

**Rationale:** Multiple databases or files add complexity. Single database simplifies backup, migration, and transaction management.

### Why Separate Tables for Different Concerns
- `threads`: Conversation threads (id, chat_id, name, created_at)
- `messages`: Message history (id, thread_id, role, content, created_at)
- `memory`: Autonomous memory (id, layer, content, created_at, updated_at)
- `model_status`: Model throttle tracking (model_id, throttled_until, last_success, failure_count)
- `kv_cache`: Key-value cache (key, value, expires_at)
- `tool_cache`: Tool result caching (tool_name, params_hash, result, expires_at)
- `channel_history`: Daily channel metrics (date, views, watch_time, subscribers)

**Rationale:** Different data types have different access patterns and schemas. Separate tables enable efficient queries and clear data modeling.

### The Threading Lock Decision
- SQLite connections are not thread-safe by default
- `_conn_lock = threading.Lock()` in db.py
- All database operations acquire lock before accessing `_conn`
- Ensures thread-safe access across async operations

**Rationale:** PrivyBot uses async/await with Telegram Bot API. Multiple concurrent operations could access SQLite simultaneously. Lock prevents race conditions.

## Consequences
### Positive
- No external service dependency
- Simple deployment (single file)
- ACID guarantees for data integrity
- Rich SQL querying capability
- Easy backup and migration
- Thread-safe with lock

### Negative
- Limited write concurrency (single connection with lock)
- No distributed scaling (single file)
- SQLite has limitations vs Postgres (no complex joins, limited data types)
- File-based I/O can be slower than in-memory

### Mitigations
- Single-user bot doesn't need write concurrency
- Lock ensures thread safety without performance impact
- SQLite limitations are acceptable for use case
- File I/O is negligible for this scale

## Alternatives Considered
**Postgres:** Rejected due to external service requirement and setup complexity.

**Redis:** Rejected due to external service requirement and volatility without persistence.

**Files:** Rejected due to lack of ACID guarantees and querying capability.

**Multiple SQLite databases:** Rejected due to added complexity with no benefit.
