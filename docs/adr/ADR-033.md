## ADR-033: MCP Compatibility

**Status:** In Progress

**Context:**
Robert's workflow operates three systems in parallel: Claude (architect) → Windsurf (executor) → PrivyBot (operator). No direct connection exists between Claude and PrivyBot. During architecture sessions, Robert manually checks stats, copies context, and bridges the gap. MCP (Model Context Protocol) — Anthropic's open protocol — closes that loop. PrivyBot's 46 tools become callable from Claude directly.

**Decision:**

**Direction:** PrivyBot as MCP server — exposing tools TO Claude. Consuming MCP servers FROM PrivyBot is a separate future concern.

**Process model:** Standalone process, not embedded in the bot. MCP server lifecycle is independent of the Telegram bot. A bot crash should not kill Claude's tool access. A separate `uv run python infra/mcp/server.py` starts the server independently.

**Transport:** Dual transport support from a single codebase:
- `--transport stdio` — Claude Desktop, local, no auth (immediate use)
- `--transport sse` — FastAPI SSE, remote, JWT auth (Tower deployment)

Transport selected at startup via command-line flag. Same tool definitions, same execution logic, different transport layer.

**Tool scope:** Curated MCP_EXPOSED_TOOLS set in `infra/mcp/config.py`. Manual configuration (not marker classes) — faster to build, clean upgrade path. Excludes internal tools (think, name_thread, audit_repo_compliance). Includes memory, calendar, email, itch, YouTube, blog, reddit, games.

**Single source of truth:** TOOL_REGISTRY remains the only definition. One converter function transforms OpenAI JSON Schema → MCP inputSchema at server startup. No duplicate tool definitions anywhere.

**Converter logic (mechanical):**
```python
# From TOOL_REGISTRY entry:
{
  "name": "get_channel_summary",
  "description": "...",
  "parameters": { "type": "object", "properties": {...} }
}

# To MCP tool:
mcp.Tool(
  name="get_channel_summary",
  description="...",
  inputSchema={ "type": "object", "properties": {...} }
)
```

**Claude Desktop configuration (stdio):**
```json
{
  "mcpServers": {
    "privybot": {
      "command": "uv",
      "args": ["run", "python", "infra/mcp/server.py", "--transport", "stdio"],
      "cwd": "C:/Github/PrivyBot"
    }
  }
}
```

**Remote access (SSE + Tailscale):**
```bash
uv run python infra/mcp/server.py --transport sse --port 8090
tailscale funnel --bg 8090
```
Public HTTPS URL: `https://nitro.ts.net/mcp`. JWT middleware validates Bearer tokens on SSE path only.

**Auth (SSE only):**
- `api/mcp/auth.py` — JWT token generation/validation using PyJWT
- `MCP_JWT_SECRET` in `.env` — secret key for signing/validating tokens
- `/mcp_token` Telegram command — generates short-lived tokens (15m/1h/24h)
- JWT middleware on SSE transport path only (stdio has no auth)

**What this enables:** From any Claude conversation — "what are my YouTube stats this week?" → calls `get_channel_summary` directly against live data. Architecture sessions gain live data without context switching. The three-system loop closes.

**Claude memory pipeline (key use case):**
Once MCP is live, Claude does not need batch export scripts to grow PrivyBot's knowledge base. During any architecture session, Claude identifies key facts and calls PrivyBot's memory write tools directly via MCP:

  save_memory("VoidDrift Phase 5 Sprint 5 scope: zoom/scroll...")
  save_memory("Robert's proof standard: raw terminal output only...")

The pre_dump_memories.py script bootstraps historical exports. MCP becomes the ongoing ingestion channel — organic, continuous, no manual steps.

Memory write tools must be included in the exposed tool set:
  save_memory, update_memory, retire_memory, get_memories

**Implementation order:**
1. Update ADR-033 (this document)
2. `uv add mcp PyJWT`
3. Add `MCP_JWT_SECRET` to `.env` and `.env.example`
4. `infra/mcp/config.py` — MCP_EXPOSED_TOOLS curated set
5. `api/mcp/auth.py` — JWT token generation/validation
6. `infra/mcp/server.py` — unified MCP server with dual transports
   - Read MCP SDK docs/source before implementing transport layer
   - Tool execution wrapper with async/sync handling (inspect.iscoroutine())
7. `/mcp_token` Telegram command
8. Claude Desktop config snippet in `README.md`
9. Tests: 6 new tests (318 + 6 = 324 total)
10. Tailscale Funnel docs

**Consequences:**

*Positive:* Claude gains live tools in every conversation (local via stdio, remote via SSE). TOOL_REGISTRY is single source of truth — tools defined once, work everywhere. Zero impact on bot operation. Architecture sessions have live data. Remote access from anywhere via Tailscale.

*Negative:* Second process to start when Claude access is wanted. Write tools (save_commitment, add_personal_task) execute against the live DB when called from Claude — intentional but explicit. Requires JWT secret management for remote access.

*Mitigations:* Server start is one command, can be added to Windows startup. Write tool side effects are covered by 204/204 tests. JWT secret is env-local, never committed. stdio path has no auth (local-only security boundary).
