# ADR-001: API/Tool Separation Pattern

## Status
Accepted

## Context
The `tools/` directory grew rapidly during PrivyBot development, leading to mixed responsibilities:
- `tools/youtube.py` contains YouTube Analytics API client code, OAuth credential handling, and tool logic
- `tools/games/metrics.py` mixes Steam API calls, SteamSpy calls, YouTube search calls, fuzzy matching, scoring, and verdict logic
- `tools/recommendations.py` duplicates some functionality from `tools/games/metrics.py`

This coupling makes it difficult to:
- Test API calls independently from tool logic
- Reuse API clients across multiple tools
- Add new tools without duplicating API integration code
- Maintain clear boundaries between data fetching and data processing

## Decision
Separate API clients from tool logic into two distinct layers:

### API Layer (`api/`)
**Responsibilities:**
- Raw API communication (HTTP requests, authentication)
- API-specific error handling and retry logic
- Response parsing and normalization
- Credential management
- Request/response logging

**What it does NOT do:**
- Tool-specific business logic
- Scoring or ranking algorithms
- Verdict generation
- Fuzzy matching or search logic
- Data transformation for tool output

**Structure:**
```
api/
  youtube_api.py    # YouTube Analytics + Data API
  steam_api.py      # Steam Web API
  steamspy_api.py   # SteamSpy API
  itad_api.py       # IsThereAnyDeal API
```

**Function contract:**
```python
def api_function(params: dict) -> dict:
    """
    Returns raw API response as dict.
    Handles auth, errors, retries.
    Never raises — returns error dict on failure.
    """
```

### Tool Layer (`tools/tools/`)
**Responsibilities:**
- Business logic and domain-specific processing
- Scoring, ranking, and verdict generation
- Fuzzy matching and search logic
- Data transformation for tool output
- Combining data from multiple API calls

**What it does NOT do:**
- Direct HTTP requests to external APIs
- API authentication
- Raw response parsing

**Structure:**
```
tools/tools/
  channel_tools.py         # Channel metrics tools
  game_tools.py            # Game metrics tools
  recommendation_tools.py  # Recommendation tools
```

**Function contract:**
```python
def tool_function(params: dict) -> dict:
    """
    Imports from api/ only.
    Applies business logic to API responses.
    Returns shaped tool response.
    Never raises — returns error dict on failure.
    """
```

## Consequences
### Positive
- Clear separation of concerns: API layer knows APIs, tool layer knows business logic
- API clients can be reused across multiple tools
- Easier to test: mock API layer for tool tests, mock HTTP for API tests
- Adding new tools is faster: import existing API clients
- Easier to maintain: API changes isolated to api/ layer

### Negative
- More files to manage (increased directory depth)
- Initial refactoring effort to separate existing code
- Slight overhead of importing from api/ vs direct calls

### Mitigations
- Keep API functions simple and focused
- Use clear naming conventions (e.g., `youtube_api.get_channel_data`)
- Document API function contracts in docstrings
- Keep tool functions focused on single responsibilities

## Examples

### Before (mixed responsibilities)
```python
# tools/youtube.py
def get_channel_summary(days: int = 7) -> dict:
    # OAuth credential handling
    creds = _get_credentials()
    # API client construction
    youtube = build("youtubeAnalytics", "v2", credentials=creds)
    # API call
    response = youtube.reports().query(...).execute()
    # Business logic
    rows = response.get("rows", [[0, 0, 0]])
    return {"views": int(row[0]), ...}
```

### After (separated)
```python
# api/youtube_api.py
def get_channel_data(days: int = 7) -> dict:
    """Raw YouTube Analytics API call."""
    creds = _get_credentials()
    youtube = build("youtubeAnalytics", "v2", credentials=creds)
    response = youtube.reports().query(...).execute()
    return {"raw": response}

# tools/tools/channel_tools.py
from tools.api.youtube_api import get_channel_data

def get_channel_summary(days: int = 7) -> dict:
    """Tool: Channel metrics with business logic."""
    api_response = get_channel_data(days)
    rows = api_response["raw"].get("rows", [[0, 0, 0]])
    return {"views": int(row[0]), ...}
```
