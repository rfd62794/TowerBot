# ADR-012: Morning Briefing Design

## Status
Accepted

## Context
PrivyBot needs a daily morning briefing to provide YouTube channel metrics. Key requirements:
- Daily channel performance summary
- Automatic delivery at consistent time
- Anomaly detection for unusual patterns
- Data persistence for historical tracking

Options considered:
- **LLM-generated every day**: Use AI to generate briefing text daily
- **Template-based standard briefing**: Use fixed template for normal days, LLM for anomalies
- **Fully manual**: User requests briefing on demand
- **No briefing**: Rely on user to check metrics manually

## Decision
Use template-based standard briefing with LLM-generated anomaly analysis.

### Why No LLM for Standard Briefing
- **Cost**: LLM calls cost money (even free models have rate limits)
- **Latency**: LLM calls add delay to briefing generation
- **Consistency**: Template ensures consistent format every day
- **Sufficiency**: Standard metrics don't require AI interpretation

**Rationale:** Standard metrics (views, watch time, subscribers) are straightforward. Template is faster, cheaper, and more consistent. LLM is only needed for anomaly analysis which requires interpretation.

### Why LLM Only on Anomaly
- **Interpretation**: Anomalies require context and interpretation
- **Insight**: LLM can provide actionable insights on unusual patterns
- **Value**: AI adds value only when data is non-standard
- **Efficiency**: Skip LLM when data is normal

**Rationale:** Normal data is self-explanatory. Anomalies (sudden drops, spikes, unusual patterns) benefit from AI interpretation. LLM is used where it adds unique value.

### Why 7AM Specifically
- **Morning routine**: 7AM aligns with typical morning check-in
- **Consistency**: Same time every day builds habit
- **Time zone**: Local time for user (configured in scheduler)
- **Before work**: Allows user to act on insights before work day

**Rationale:** 7AM is early enough to be actionable but not so early as to be disruptive. Consistent timing builds user habit.

### Why channel_history Accumulates Daily
- **Historical tracking**: Daily accumulation enables trend analysis
- **Comparison**: Compare today's metrics to previous days
- **Anomaly detection**: Need historical baseline to detect anomalies
- **Persistence**: Data persists in SQLite channel_history table

**Rationale:** Without daily accumulation, anomaly detection is impossible. Historical data provides baseline for comparison.

### The Data Persistence Rationale
- **SQLite channel_history table**: Stores daily metrics (date, views, watch_time, subscribers)
- **Automatic recording**: morning_briefing in core/scheduler.py records data daily
- **Idempotent**: INSERT OR REPLACE ensures no duplicates on re-runs
- **Queryable**: Historical data can be queried for trends and comparisons

**Rationale:** Persistent historical data enables anomaly detection and trend analysis. SQLite provides reliable persistence without external service.

## Consequences
### Positive
- Cost-effective (LLM only when needed)
- Fast standard briefing (no LLM latency)
- Consistent format (template-based)
- Actionable anomaly insights (LLM interpretation)
- Historical tracking for trend analysis

### Negative
- Template may become stale over time
- Anomaly detection requires threshold tuning
- 7AM may not align with all schedules
- Historical data grows unbounded

### Mitigations
- Update template periodically based on user feedback
- Tune anomaly thresholds based on actual data patterns
- Allow configurable briefing time
- Implement data retention policy if needed

## Alternatives Considered
**LLM-generated every day:** Rejected due to cost, latency, and lack of added value for standard metrics.

**Fully manual:** Rejected due to lack of automation and consistency.

**No briefing:** Rejected due to lack of visibility into channel performance.

**No historical tracking:** Rejected due to inability to detect anomalies or trends.
