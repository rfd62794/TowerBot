# ADR-009: Python 3.12 + uv

## Status
Accepted

## Context
PrivyBot requires a Python runtime and dependency management. Initial attempt used Python 3.14, but encountered a critical bug.

### The Python 3.14 Bug
- Error chain: Python 3.14 → httpx → httpcore → anyio → start_tls → BrokenResourceError
- Default ProactorEventLoop on Windows intermittently resets async TLS handshakes to api.telegram.org
- Error: `BrokenResourceError` and `ConnectError` on new outbound handshakes
- Warm pooled connections (getUpdates) survive, but new handshakes (send_chat_action, send_message) fail intermittently
- Workaround: SelectorEventLoop fixed most issues but one transient failure remained
- Final solution: Downgraded to Python 3.12 via uv

### Dependency Management Options
- **pip + venv**: Standard Python tooling
- **poetry**: Modern dependency manager with lock files
- **uv**: Fast Rust-based package manager, compatible with pip

## Decision
Use Python 3.12 with uv for dependency management.

### Why Not 3.14 (The Bug We Hit)
- Python 3.14 has a breaking change in async TLS handling on Windows
- httpx/anyio libraries fail to complete TLS handshakes reliably
- ProactorEventLoop (default on Windows) causes intermittent connection failures
- Workaround (SelectorEventLoop) is fragile and may not work in future versions
- No upstream fix available at time of implementation

**Rationale:** The bug is critical for Telegram Bot API integration. Python 3.12 is the latest stable version without this issue.

### Why uv Over pip/venv
- **Speed**: uv is significantly faster than pip (Rust-based)
- **Compatibility**: uv is a drop-in replacement for pip, compatible with existing workflows
- **Lock files**: uv generates lock files for reproducible builds
- **Modern**: uv is actively maintained and designed for modern Python workflows

**Rationale:** uv provides better performance and modern features while maintaining compatibility with pip. No migration required from existing pip-based projects.

### Why Pinned Version in pyproject.toml
- `requires-python = ">=3.12,<3.13"` in pyproject.toml
- Explicitly pins to Python 3.12 only
- Prevents accidental upgrade to 3.14
- Ensures reproducible builds

**Rationale:** Pinning prevents accidental upgrade to broken version. uv enforces this constraint.

## Consequences
### Positive
- Avoids Python 3.14 TLS bug
- Fast dependency management with uv
- Reproducible builds with lock files
- Compatible with existing pip workflows

### Negative
- Locked to Python 3.12 until 3.14 bug is fixed
- uv is newer tool, less battle-tested than pip
- Pinning prevents automatic upgrades

### Mitigations
- Monitor Python 3.14 for TLS fix
- uv is gaining adoption, actively maintained
- Pinning can be relaxed when bug is fixed
- uv is compatible with pip, easy to switch if needed

## Alternatives Considered
**Python 3.14 with SelectorEventLoop workaround:** Rejected due to fragility and potential future breakage.

**pip + venv:** Rejected due to slower performance and lack of lock files.

**poetry:** Rejected due to complexity and slower performance than uv.

**No version pinning:** Rejected due to risk of accidental upgrade to broken 3.14.
