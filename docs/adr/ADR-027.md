# ADR-027: PollingManager Layer

## Status
Accepted

## Context

### Current State
- Heartbeat fires every 60min for all data fetching regardless of individual TTLs
- Gmail TTL is 5min but heartbeat is 60min — data can be 55min stale at briefing
- `_initial_sync` in privybot.py and `scripts/preload.py` overlap with heartbeat data fetching
- No single owner of "when to poll next"
- CacheManager owns "what data do we have"
- RateLimitManager owns "can we call now"
- Nothing owns "when do we call next"

### Problems
1. **Inefficient polling:** All data sources polled at same 60min interval regardless of data staleness
2. **Stale data at critical moments:** Gmail (5min TTL) polled at 60min intervals means briefing reads 55min-old data
3. **Duplicate fetching:** preload.py, _initial_sync, and heartbeat all fetch similar data on startup
4. **No coordination:** No awareness of cache freshness before polling
5. **No rate limit awareness:** Polling doesn't check RateLimitManager before firing

## Decision

### Solution
Create `infra/polling.py` with `PollingManager` class — Layer 7c, parallel to CacheManager (7) and RateLimitManager (7b).

### Architecture
```
Layer 7 cluster — Infrastructure Services

CacheManager   — answers: "what data do we have?"
RateLimitManager — answers: "can we call now?"
PollingManager — answers: "when do we call next?"
```

### Design
```python
# infra/polling.py

class PollingManager:
    """
    Single owner of all polling behavior.
    Replaces scattered heartbeat checks.
    Each data source has its own interval.
    Coordinates with Cache + RateLimit.
    """
    
    # Default poll intervals — match TTLs from CacheManager exactly
    INTERVALS = {
        "gmail_personal":   300,   # 5min
        "gmail_rfd":        300,   # 5min
        "calendar_today":   900,   # 15min
        "google_tasks":     300,   # 5min
        "youtube_channel":  86400, # 24h
        "steam_library":    86400, # 24h
        "weather":          3600,  # 1h
        "ddg_search":       None,  # on-demand only
        "wikipedia":        None,  # on-demand only
        "reddit":           None,  # on-demand only
    }
    
    def register(self,
                 key: str,
                 fn: callable,
                 interval: int = None,
                 args: dict = None) -> None:
        """
        Register a data source for polling.
        interval: seconds between polls. None = on-demand only.
        args: kwargs to pass to fn.
        """
    
    def is_due(self, key: str) -> bool:
        """
        Is this poll due to run?
        Checks last_polled_at against interval.
        Also checks CacheManager — if cache is still fresh, skip the poll.
        """
    
    def run_poll(self, key: str) -> dict:
        """
        Execute one poll.
        Checks RateLimitManager first.
        Records result in poll_log.
        Updates CacheManager via the fn.
        """
    
    async def run_loop(self) -> None:
        """
        Main async loop.
        Checks every 60s which polls are due.
        Fires due polls sequentially.
        Replaces heartbeat polling logic.
        """
    
    def status(self) -> list[dict]:
        """
        Current state of all registered polls.
        Used by /status command.
        Shows: last_polled, next_due, success.
        """

polling_manager = PollingManager()
```

### Database Schema
```sql
CREATE TABLE IF NOT EXISTS poll_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    poll_key TEXT NOT NULL,
    polled_at TEXT NOT NULL,
    success INTEGER DEFAULT 1,
    duration_ms INTEGER,
    from_cache INTEGER DEFAULT 0,
    error_msg TEXT
)

CREATE INDEX IF NOT EXISTS idx_poll_log_key_time
    ON poll_log(poll_key, polled_at DESC)
```

### Absorbs
- `scripts/preload.py` — startup preload logic
- Heartbeat data fetching (10 checks)
- `_initial_sync` in privybot.py

### Keeps in Scheduler
- Morning briefing send
- Nightly summary send
- Alert detection (overdue tasks, events)
- Health checks (deploy status, DB)

### Concurrency Note
PollingManager runs its own async loop parallel to scheduler. Risk: briefing fires at 07:00:01 while gmail poll fires at 07:00:00. Briefing reads cache before poll completes, gets slightly stale data.

**Fix:** `PollingManager.wait_for(key)` method for briefing to wait for in-progress polls before reading cache.

## Consequences

### Positive
- Each data source polls at correct rate (Gmail at 5min not 60min)
- Briefing always reads warm cache
- No duplicate fetching between preload and heartbeat
- Single point of polling control
- Visible in `/status` command
- Coordinates with CacheManager (skip if fresh) and RateLimitManager (check before call)

### Negative
- Adds complexity: two concurrent async loops need coordination
- Requires `wait_for(key)` mechanism for briefing-poll race condition
- Additional DB table (poll_log)

### Mitigations
- Clear separation: Scheduler owns sends/alerts, PollingManager owns data fetching
- Simple coordination: `wait_for(key)` with timeout
- PollingManager is optional — can run without it (fallback to current behavior)
